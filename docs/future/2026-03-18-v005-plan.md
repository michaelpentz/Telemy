# Telemy v0.0.5 Implementation Plan

**Date:** 2026-03-18
**Author:** Michael Pentz
**Status:** Draft ‚Äî active planning
**Baseline:** v0.0.4 (commit `e9ce030`, all repos clean and pushed)

---

## Vision

v0.0.4 proved the full relay lifecycle works: ephemeral AWS EC2 instances, bonded SRT/SRTLA, per-user stream tokens, EIP stability, real-time stats in the dock. It works. It's also expensive ‚Äî $9‚Äì13/user/month on AWS, with $3.60/user/month in EIP idle costs alone. At 100 users that's ~$912/month in infrastructure before a single dollar of revenue.

v0.0.5 is the cost revolution. The core shift: from AWS-only per-user ephemeral relays to a flexible model supporting two paths:

1. **BYOR (Bring Your Own Relay)** ‚Äî free tier. Users point the plugin at their own SRTLA endpoint. Zero infrastructure cost to Telemy. The plugin becomes a monitoring/telemetry/scene-management overlay.
2. **Managed shared relays on Hetzner Cloud** ‚Äî paid tier. Always-on shared VPS instances replace ephemeral per-user EC2. Multiple users share a relay server via per-user stream tokens (already implemented in v0.0.4). Target cost: ~$0.50/user/month.

v0.0.4 is preserved as the AWS-backed release and fallback. The `AWSProvisioner` stays in the codebase, selectable via `AEGIS_RELAY_PROVIDER=aws`, but it's no longer the default path.

The control plane itself also migrates off AWS EC2 to Hetzner, cutting the baseline hosting cost from EC2 pricing to ~$5/month.


Beyond the cost and infrastructure work in Phases 1-5, Telemy's longer-term roadmap includes a **real-time video stabilization** feature (Phase 6) ó a GPU-accelerated OBS filter for IRL streamers that no competitor currently offers. This is planned for after the relay infrastructure is solid, and will position Telemy as the complete IRL streaming toolkit, not just a relay manager. See Phase 6 below and docs/future/video-stabilization.md for the full design.

## Key Decisions

| Decision | Rationale |
|----------|-----------|
| BYOR mode is free (no account required for relay, account optional for telemetry sync) | No infra cost to Telemy; plugin is already cloud-agnostic; lowers barrier to adoption |
| Managed relays move from AWS EC2 on-demand to shared always-on Hetzner Cloud VPS | 94‚Äì97% cost reduction; 20TB included traffic eliminates overage risk; no per-user idle cost |
| AWS provisioner code stays in codebase as backup/premium option | Proven, tested, zero-risk fallback; could be future "dedicated" tier |
| Control plane migrates from EC2 (`52.13.2.122`) to Hetzner (same or co-located box) | Consolidate infra on one provider; ~$5/mo vs current EC2 cost |
| Per-user stream tokens handle multi-tenant security on shared relays | Already implemented in v0.0.4 (`live_{stream_token}` / `play_{stream_token}`); no new auth model needed |
| Tag v0.0.4 as stable release before any v0.0.5 changes begin | Clean rollback point; v0.0.5 development on a new branch |

## Current State (What We're Building On)

### Provisioner Interface (Already Abstract)

From `internal/relay/provisioner.go`:

```go
type Provisioner interface {
    Provision(ctx context.Context, req ProvisionRequest) (ProvisionResult, error)
    Deprovision(ctx context.Context, req DeprovisionRequest) error
}
```

This is the primary abstraction seam. `FakeProvisioner` already proves it works. The interface is clean ‚Äî the problem is everything around it.

### What's Already Provider-Agnostic

- Session state machine (`provisioning` ‚Üí `active` ‚Üí `grace` ‚Üí `stopped`)
- Plugin relay client (only knows HTTP endpoints and IPs, zero cloud awareness)
- Docker relay stack (runs anywhere Docker runs)
- Entitlement system (plan tiers and usage tracking)
- Stats polling (`relay_client.cpp` polls `:8090` and `:5080` ‚Äî works with any host)
- DNS management (Cloudflare, called from pipeline but not coupled to provider)

### What's Tightly Coupled to AWS (Needs Refactoring)

| Coupling | Location | Fix |
|----------|----------|-----|
| `AWSInstanceID` field in `ProvisionResult` | `provisioner.go` | Rename to `InstanceID` |
| `AWSInstanceID` field in `DeprovisionRequest` | `provisioner.go` | Rename to `InstanceID` |
| `aws_instance_id` column in `relay_instances` table | SQLite schema | Rename to `instance_id` |
| EIP type-assertion in `runProvisionPipeline()` | `handlers.go` ~line 200 | Extract into provider interface or conditional hook |
| User-data script embedded as Go string constant | `aws.go` | Make template parameter |

| AMI map config (`AEGIS_AWS_AMI_MAP`) | `config.go` | Keep but don't require when provider != aws |

**Validation:** Rebuild control plane binary, run against existing SQLite DB with migration, verify relay start/stop cycle works unchanged.

### 1.2 Extract EIP Logic from Provision Pipeline

**Estimated effort:** 2‚Äì3 hours

Current problem in `handlers.go`:
```go
if awsProv, ok := s.provisioner.(*relay.AWSProvisioner); ok {
    // EIP allocation/association logic
}
```

This breaks the `Provisioner` abstraction. Fix:

**Option A (preferred):** Add an optional `StaticIPManager` interface:
```go
type StaticIPManager interface {
    EnsureStaticIP(ctx context.Context, userID string, instanceID string) (string, error)
    ReleaseStaticIP(ctx context.Context, userID string) error
}
```

The `AWSProvisioner` implements both `Provisioner` and `StaticIPManager`. The pipeline checks for the interface:
```go
if ipMgr, ok := s.provisioner.(relay.StaticIPManager); ok {
    ip, err := ipMgr.EnsureStaticIP(ctx, userID, result.InstanceID)
    // ...
}
```

**Option B:** Move EIP handling entirely inside `AWSProvisioner.Provision()`, so the pipeline never sees it.

Option A is cleaner because it keeps `Provision()` focused on the instance lifecycle and makes static IP management an explicit opt-in capability.

**Validation:** Same relay start/stop cycle. EIP behavior unchanged.

**Estimated effort:** 2‚Äì3 hours

The template parameters: `StreamToken`, `SRTLAPort`, `SRTPlayerPort`, `StatsPort`, `PerLinkStatsPort`, `ManagementPort`, `DockerImage`.

## Phase 3: Hetzner Shared Relay Provisioner (Week 2‚Äì3)

**Goal:** Managed relay users get assigned to shared always-on Hetzner VPS instances instead of ephemeral per-user EC2 instances. Cost drops from ~$9‚Äì13/user/month to ~$0.50/user/month.

### 3.1 Relay Pool Schema

**Estimated effort:** 0.5 days

New migration `0010_relay_pool.sql`:
```sql
CREATE TABLE relay_pool (
    server_id       TEXT PRIMARY KEY,          -- Hetzner server ID or manual identifier
    provider        TEXT NOT NULL DEFAULT 'hetzner',
    host            TEXT NOT NULL,             -- hostname or IP
    ip              TEXT NOT NULL,             -- public IPv4
    region          TEXT NOT NULL,             -- e.g. 'us-west', 'us-east', 'eu-central'
    status          TEXT NOT NULL DEFAULT 'active',  -- active, draining, offline
    current_sessions INTEGER NOT NULL DEFAULT 0,
    max_sessions    INTEGER NOT NULL DEFAULT 20,
    created_at      DATETIME DEFAULT CURRENT_TIMESTAMP,
    last_health_check DATETIME,
    health_status   TEXT DEFAULT 'unknown'     -- healthy, degraded, unhealthy, unknown
);

CREATE TABLE relay_assignments (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    user_id         TEXT NOT NULL,
    session_id      TEXT NOT NULL,
    server_id       TEXT NOT NULL REFERENCES relay_pool(server_id),
    stream_token    TEXT NOT NULL,
    assigned_at     DATETIME DEFAULT CURRENT_TIMESTAMP,
    released_at     DATETIME,
    UNIQUE(session_id)
);
```

The `relay_pool` table is manually populated ‚Äî Hetzner servers are provisioned once (manually or via a setup script), then registered in the pool. This is NOT dynamic auto-scaling; it's a fixed pool that grows by manually adding servers.

### 3.2 HetznerProvisioner

**Estimated effort:** 3‚Äì4 days

```go
type HetznerProvisioner struct {
    store     *store.Store
    sls       *SLSClient  // HTTP client for SLS management API
}

func (p *HetznerProvisioner) Provision(ctx context.Context, req ProvisionRequest) (ProvisionResult, error) {
    // 1. Find an available relay server in the pool
    server, err := p.store.AssignRelay(req.UserID, req.SessionID, req.Region, req.StreamToken)
    // AssignRelay picks: region-affinity first, then least-loaded, then any available
    
    // 2. Create user's stream IDs on the shared relay via SLS API
    err = p.sls.CreateStreamIDs(server.Host, req.StreamToken)
    // POST http://{server.host}:8090/api/stream-ids
    // Creates live_{stream_token} and play_{stream_token}
    
    // 3. Return the shared server's connection details
    return ProvisionResult{
        InstanceID:   server.ServerID,
        PublicIP:     server.IP,
        SRTPort:      5000,
    }, nil
}

func (p *HetznerProvisioner) Deprovision(ctx context.Context, req DeprovisionRequest) error {
    // 1. Remove user's stream IDs from the relay
    err := p.sls.DeleteStreamIDs(server.Host, streamToken)
    
    // 2. Release the assignment (decrement current_sessions)
    err = p.store.ReleaseRelay(req.SessionID)
    
    // 3. Do NOT destroy the server ‚Äî it stays in the pool
    return nil
}
```

**Key difference from AWS flow:**
- No instance launch/terminate. The server is already running.
- No EIP allocation. The shared server has a fixed IP.
- No user-data script. The Docker stack is pre-deployed.
- Provision is fast (HTTP call to SLS API, <1 second) vs AWS (60‚Äì90 seconds for EC2 boot).
- The provision step tracking (`launching_instance` ‚Üí `waiting_for_instance` ‚Üí ...) can be simplified or skipped.

This client is also useful for the health monitor (3.4).

### 3.4 Pool Health Monitor

**Estimated effort:** 1 day

Background goroutine that runs on a timer (every 60 seconds):
- Iterates over all `active` servers in `relay_pool`
- Calls `SLSClient.HealthCheck()` for each
- Updates `last_health_check` and `health_status`
- If a server is `unhealthy` for >3 consecutive checks, marks it `draining` (no new assignments)
- Alerts (log + optional webhook) if a server goes offline

### 3.5 Assignment Logic

**Estimated effort:** 0.5 days

`store.AssignRelay(userID, sessionID, region, streamToken)`:

1. Query `relay_pool` where `status = 'active'` and `current_sessions < max_sessions` and `health_status = 'healthy'`
2. Filter by region affinity: prefer same region as user's request
3. Sort by `current_sessions ASC` (least-loaded first)
4. If no servers available in preferred region, fall back to any region
5. If no servers available at all, return `ErrNoRelayCapacity`
6. Atomically: increment `current_sessions`, insert `relay_assignments` row
7. Return the selected server

### 3.6 Provision Pipeline Adaptation

**Estimated effort:** 0.5 days

The existing `runProvisionPipeline()` in `handlers.go` runs 6 steps with 3-second dwells. For Hetzner shared relays, most of those steps are instant:

- `launching_instance` ‚Üí skip (server already running)
- `waiting_for_instance` ‚Üí skip
- `starting_docker` ‚Üí skip (containers already running)
- `starting_containers` ‚Üí skip
- `creating_stream` ‚Üí this is the only real step (SLS API call)
- `ready` ‚Üí done

Options:
- **Option A:** Fast-path the pipeline: if provider is Hetzner, jump straight from `creating_stream` to `ready` with no artificial dwells.
- **Option B:** Keep the step progression but with 0-second dwells, so the dock UI flashes through quickly. This gives a consistent UX.

**Recommendation:** Option A. Users will appreciate the instant start.

### 3.7 No Per-User EIP

Shared Hetzner relays have a fixed IP. No per-user EIP needed. The `StaticIPManager` interface from Phase 1 simply isn't implemented by `HetznerProvisioner` ‚Äî the pipeline skips it automatically.

DNS records (`{slug}.relay.telemyapp.com`) point to the shared relay's IP. Multiple users can share the same relay IP because stream isolation is handled by unique stream tokens, not by separate IPs.

### 3.8 Phase 3 Deliverable

At the end of Phase 3, a paid user can:
1. Click "Start Relay" in the dock
2. Get assigned to a shared Hetzner relay in <2 seconds (vs 60‚Äì90s on AWS)
3. Stream via `srtla://{relay_ip}:5000` with `streamid=live_{stream_token}`
4. See full stats in the dock (same as v0.0.4)
5. Click "Stop Relay" ‚Äî stream IDs are cleaned up, server stays running

## Phase 6: Video Stabilization (Future ó After Phases 1-5)

Real-time video stabilization as an optional OBS filter for IRL streamers. This is a differentiator ó no competitor offers this.

Full design doc: `docs/future/video-stabilization.md`

### Summary

Software-based stabilization running on the receiving PC's GPU (compute shaders via Rust + wgpu), processing the incoming SRT/SRTLA feed after arrival and before OBS encode. Local-only ó cloud-hosted stabilization was evaluated and rejected (latency, bandwidth, cost).

### Key Design Decisions

- **Build fresh in Rust with wgpu** ó do NOT fork LiveVisionKit (OpenCV/OpenCL driver issues, different codebase conventions). Use LVK as algorithm/UX reference only.
- **Completely decoupled** from core relay/failover logic ó independent optional filter
- **Hardware gated** ó minimum RTX 2060 / RX 5000, runtime VRAM and compute utilization checks, auto-disable above ~75% GPU compute usage
- **Configurable crop factor** ó slider 1-15%, sensible defaults (3-5% handheld, 8-10% walking)
- **Audio sync** ó Rust core handles automatic delay compensation for frame buffer latency
- **Suppression mode** ó auto-disables during intentional rapid panning

### Processing Pipeline

```
Incoming frame (SRT/SRTLA source)
  ? Motion Estimation (compute shader)
  ? Smoothing Buffer (N frames)
  ? Corrective Transform (compute shader)
  ? Crop & Scale (+ optional FSR sharpening)
  ? Output ? OBS encode
```

### Resource Impact

- Uses GPU compute cores (NOT the hardware encoder ó NVENC/AMF runs in parallel)
- ~200-400MB VRAM at 1080p
- Excellent on dedicated encoding PCs (RTX 2060+), good with dual H/V output
- Not recommended for GTX 1060 or below

### Implementation Effort

- Estimated 4-6 weeks for MVP (motion estimation + basic smoothing + crop)
- Prototype first: minimal wgpu compute shader for frame-to-frame motion estimation
- Benchmark across GPU generations before committing to approach

### Open Questions (Resolve at Implementation Time)

1. Motion estimation algorithm: optical flow vs phase correlation vs block matching
2. OBS integration point: register as OBS filter or process within aegis-core
3. Gyroscope metadata from phones for hybrid stabilization
4. Interaction with OBS scene transforms and crop filters

### Why This Matters for Telemy

IRL streamers are the core audience. Shaky footage is their #1 quality complaint. Current solutions (gimbals, in-camera EIS) all have gaps. A bundled stabilizer in the Telemy plugin ó alongside failover, relay, telemetry ó makes Telemy the complete IRL toolkit, not just a relay manager.

### Pre-Migration (Before any v0.0.5 code changes)

1. **Tag v0.0.4:** `git tag v0.0.4-stable` on all three repos at current HEADs
2. **Branch:** Create `v0.0.5-dev` branch from `main`
3. **Document:** Snapshot current EC2 instance state, EIP allocations, DNS records, AMI versions

### Development Sequence

All v0.0.5 work happens on the `v0.0.5-dev` branch. Merges to `main` only after each phase is validated.

### Rollback Plan

- If Hetzner relays have issues ‚Üí switch `AEGIS_RELAY_PROVIDER=aws` and restart
- If control plane migration fails ‚Üí revert DNS to `52.13.2.122` (EC2 stays running for 48h)
- If BYOR mode has bugs ‚Üí disable the free tier endpoint, users fall back to managed
- At no point is v0.0.4 AWS functionality removed ‚Äî it's deactivated, not deleted

### Existing User Preservation

- Existing user EIPs (`eipalloc-05476ee61b32e4891` etc.) remain allocated
- Existing DNS records (`8ae1cf.relay.telemyapp.com` etc.) remain pointed at EIPs
- Existing AMIs (`aegis-relay-v2`) remain available
- If a user needs AWS-backed relay access, flip their `relay_mode` to `dedicated`

## Risk Assessment

| Risk | Severity | Likelihood | Mitigation |
|------|----------|------------|------------|
| BYOR mode: plugin works but stats don't (non-SLS relays) | Low | Medium | Graceful degradation ‚Äî show "Stats unavailable", don't error |
| Hetzner provisioner: new API, pool management bugs | Medium | Medium | Thorough testing with FakeProvisioner first; manual pool management (no auto-scaling) |
| Control plane migration: DNS propagation issues | Medium | Low | Low TTL pre-cutover, keep EC2 running 48h, monitor both endpoints |
| Shared relay: noisy neighbor (one user's stream degrades others) | Medium | Low | Conservative `max_sessions` (10‚Äì15 per CX22), monitor per-server metrics |
| Hetzner availability/reliability | Medium | Low | Hetzner has strong SLA for cloud servers; AWS remains as backup provider |
| AWS code bitrot (untested over time) | Low | Medium | Keep AWS integration tests in CI; periodic manual validation |
| Multi-tenant stream isolation failure | High | Very Low | Per-user stream tokens are already enforced by SLS; tokens are cryptographically random |

## Open Questions

### 1. Should BYOR users need an account at all?

**Arguments for requiring an account:**
- Telemetry sync across sessions (useful for the user)
- Usage analytics for Telemy (useful for us)
- Upsell path from free to paid
- Scene preferences need server-side storage if syncing across machines

**Arguments against:**
- Lower friction for adoption
- Some users just want the OBS dock widget, no cloud dependency
- Reduces control plane load

**Proposed answer:** Account optional. Plugin works fully offline in BYOR mode (local config, local scene prefs). If the user creates an account, their BYOR config syncs to the server and they get cross-machine persistence. This is the best of both worlds.

### 2. How to handle BYOR users' relay stats if their relay doesn't run SLS?

**Proposed answer:** Graceful degradation. The plugin attempts to poll `:8090/stats` and `:5080/stats`. If either returns a connection error or non-200 status for 3 consecutive polls, that section is hidden in the dock with a subtle "Stats unavailable for custom relays" message. The rest of the dock (OBS telemetry, system stats, scene management) works regardless.

For users running Belabox: Belabox uses SLS internally, so stats *might* work if the stats port is exposed. Worth testing.

### 3. Minimum viable Hetzner regions to launch with?

**Proposed answer:** Start with two:
- **US East (Ashburn)** ‚Äî Hetzner's `ash` datacenter, covers East Coast and Central US
- **US West (Hillsboro)** ‚Äî Hetzner's `hil` datacenter, covers West Coast

Most Telemy users are US-based IRL streamers. EU can be added later with Hetzner's `fsn1` (Falkenstein) or `nbg1` (Nuremberg).

Assignment logic defaults to `us-east` if no region preference, or picks based on user's last known location.

### 4. Whether to publish the srtla-receiver Docker stack as a public self-host guide

**Proposed answer:** Yes, but after v0.0.5 launches and BYOR is validated. The Docker image is already public (`ghcr.io/michaelpentz/srtla-receiver:latest`). A simple README with:
- `docker-compose.yml` example
- Firewall rules (UDP 5000, TCP 4000, TCP 8090, TCP 5080)
- Stream ID creation via SLS API
- Telemy plugin BYOR configuration

This increases adoption (more BYOR users ‚Üí more potential conversions to paid) and positions Telemy as the ecosystem tool for IRL streaming, not just a relay provider.
