# Telemy v0.0.5 Implementation Plan

**Date:** 2026-03-18
**Author:** Michael Pentz
**Status:** Draft — active planning
**Baseline:** v0.0.4 (commit `e9ce030`, all repos clean and pushed)

---

## Vision

v0.0.4 proved the full relay lifecycle works: ephemeral AWS EC2 instances, bonded SRT/SRTLA, per-user stream tokens, EIP stability, real-time stats in the dock. It works. It's also expensive — $9–13/user/month on AWS, with $3.60/user/month in EIP idle costs alone. At 100 users that's ~$912/month in infrastructure before a single dollar of revenue.

v0.0.5 is the cost revolution. The core shift: from AWS-only per-user ephemeral relays to a flexible model supporting two paths:

1. **BYOR (Bring Your Own Relay)** — free tier. Users point the plugin at their own SRTLA endpoint. Zero infrastructure cost to Telemy. The plugin becomes a monitoring/telemetry/scene-management overlay.
2. **Managed shared relays on Hetzner Cloud** — paid tier. Always-on shared VPS instances replace ephemeral per-user EC2. Multiple users share a relay server via per-user stream tokens (already implemented in v0.0.4). Target cost: ~$0.50/user/month.

v0.0.4 is preserved as the AWS-backed release and fallback. The `AWSProvisioner` stays in the codebase, selectable via `AEGIS_RELAY_PROVIDER=aws`, but it's no longer the default path.

The control plane itself also migrates off AWS EC2 to Hetzner, cutting the baseline hosting cost from EC2 pricing to ~$5/month.


Beyond the cost and infrastructure work in Phases 1-5a, Telemy's longer-term roadmap includes a **real-time video stabilization** feature (Phase 6, targeted for **v0.0.6**) � a GPU-accelerated OBS filter for IRL streamers that no competitor currently offers. This is planned for after the relay infrastructure is solid, and will position Telemy as the complete IRL streaming toolkit, not just a relay manager. See Phase 6 below and docs/future/video-stabilization.md for the full design.

## Key Decisions

| Decision | Rationale |
|----------|-----------|
| BYOR mode is free (no account required for relay, account optional for telemetry sync) | No infra cost to Telemy; plugin is already cloud-agnostic; lowers barrier to adoption |
| Managed relays move from AWS EC2 on-demand to shared always-on Hetzner Cloud VPS | 94–97% cost reduction; 20TB included traffic eliminates overage risk; no per-user idle cost |
| AWS provisioner code stays in codebase as backup/premium option | Proven, tested, zero-risk fallback; could be future "dedicated" tier |
| Control plane migrates from EC2 (`<redacted-ec2-ip>`) to Hetzner (same or co-located box) | Consolidate infra on one provider; ~$5/mo vs current EC2 cost |
| Per-user stream tokens handle multi-tenant security on shared relays | Already implemented in v0.0.4 (`live_{stream_token}` / `play_{stream_token}`); no new auth model needed |
| Tag v0.0.4 as stable release before any v0.0.5 changes begin | Clean rollback point; v0.0.5 development on a new branch |

## Current State (What We're Building On)

### Provisioner Interface (Already Abstract)

From `internal/relay/provisioner.go`:

```go
type Provisioner interface {
    Provision(ctx context.Context, req ProvisionRequest) (ProvisionResult, error)
    Deprovision(ctx context.Context, req DeprovisionRequest) error
}
```

This is the primary abstraction seam. `FakeProvisioner` already proves it works. The interface is clean — the problem is everything around it.

### What's Already Provider-Agnostic

- Session state machine (`provisioning` → `active` → `grace` → `stopped`)
- Plugin relay client (only knows HTTP endpoints and IPs, zero cloud awareness)
- Docker relay stack (runs anywhere Docker runs)
- Entitlement system (plan tiers and usage tracking)
- Stats polling (`relay_client.cpp` polls `:8090` and `:5080` — works with any host)
- DNS management (Cloudflare, called from pipeline but not coupled to provider)

### What's Tightly Coupled to AWS (Needs Refactoring)

| Coupling | Location | Fix |
|----------|----------|-----|
| `AWSInstanceID` field in `ProvisionResult` | `provisioner.go` | Rename to `InstanceID` |
| `AWSInstanceID` field in `DeprovisionRequest` | `provisioner.go` | Rename to `InstanceID` |
| `aws_instance_id` column in `relay_instances` table | PostgreSQL schema | Rename to `instance_id` |
| EIP type-assertion in `runProvisionPipeline()` | `handlers.go` ~line 200 | Extract into provider interface or conditional hook |

| AMI map config (`AEGIS_AWS_AMI_MAP`) | `config.go` | Keep but don't require when provider != aws |

### 1.1 Rename AWS-Specific Fields

**Estimated effort:** 2–3 hours

- `DeprovisionRequest.AWSInstanceID` → `DeprovisionRequest.InstanceID`
- `relay_instances.aws_instance_id` → `relay_instances.instance_id` (new migration `0008_rename_instance_id.sql`)

**Validation:** Rebuild control plane binary, run against existing PostgreSQL DB with migration, verify relay start/stop cycle works unchanged.

### 1.2 Extract EIP Logic from Provision Pipeline

**Estimated effort:** 2–3 hours

Current problem in `handlers.go`:
```go
if awsProv, ok := s.provisioner.(*relay.AWSProvisioner); ok {
    // EIP allocation/association logic
}
```

This breaks the `Provisioner` abstraction. Fix:

**Option A (preferred):** Add an optional `StaticIPManager` interface:
```go
type StaticIPManager interface {
    EnsureStaticIP(ctx context.Context, userID string, instanceID string) (string, error)
    ReleaseStaticIP(ctx context.Context, userID string) error
}
```

The `AWSProvisioner` implements both `Provisioner` and `StaticIPManager`. The pipeline checks for the interface:
```go
if ipMgr, ok := s.provisioner.(relay.StaticIPManager); ok {
    ip, err := ipMgr.EnsureStaticIP(ctx, userID, result.InstanceID)
    // ...
}
```

**Option B:** Move EIP handling entirely inside `AWSProvisioner.Provision()`, so the pipeline never sees it.

Option A is cleaner because it keeps `Provision()` focused on the instance lifecycle and makes static IP management an explicit opt-in capability.

> **Review correction:** The entire EIP workflow (allocation, persistence to `users`, association, and orphan release) should move fully behind the AWS provider boundary -- not just extracted to an interface. Currently EIP logic is split between `handlers.go` and `aws.go`, which creates leak risk on partial failure. The `AWSProvisioner` should own the complete EIP lifecycle internally. For non-AWS providers, no static IP management interface is needed at all.

**Validation:** Same relay start/stop cycle. EIP behavior unchanged.

### 1.3 Templatize User-Data Script

**Estimated effort:** 2–3 hours

The relay bootstrap script (write docker-compose, start containers, create stream IDs) is currently an embedded Go string constant in `aws.go` with `STREAM_TOKEN` placeholder replacement.

Refactor:

The template parameters: `StreamToken`, `SRTLAPort`, `SRTPlayerPort`, `StatsPort`, `PerLinkStatsPort`, `ManagementPort`, `DockerImage`.

**Validation:** Deploy a test relay with the templatized script, verify full container startup and stream ID creation.

## Phase 3: Hetzner Shared Relay Provisioner (Week 2–3)

**Goal:** Managed relay users get assigned to shared always-on Hetzner VPS instances instead of ephemeral per-user EC2 instances. Cost drops from ~$9–13/user/month to ~$0.50/user/month.

### 3.1 Relay Pool Schema

**Estimated effort:** 0.5 days

New migration `0010_relay_pool.sql`:
```sql
CREATE TABLE relay_pool (
    server_id       TEXT PRIMARY KEY,          -- Hetzner server ID or manual identifier
    provider        TEXT NOT NULL DEFAULT 'hetzner',
    host            TEXT NOT NULL,             -- hostname or IP
    ip              INET NOT NULL,             -- public IPv4
    region          TEXT NOT NULL,             -- e.g. 'us-west', 'us-east', 'eu-central'
    status          TEXT NOT NULL DEFAULT 'active',  -- active, draining, offline
    current_sessions INTEGER NOT NULL DEFAULT 0,
    max_sessions    INTEGER NOT NULL DEFAULT 20,
    created_at      TIMESTAMPTZ DEFAULT NOW(),
    last_health_check TIMESTAMPTZ,
    health_status   TEXT DEFAULT 'unknown'     -- healthy, degraded, unhealthy, unknown
);

CREATE TABLE relay_assignments (
    id              SERIAL PRIMARY KEY,
    user_id         TEXT NOT NULL,
    session_id      TEXT NOT NULL,
    server_id       TEXT NOT NULL REFERENCES relay_pool(server_id),
    stream_token    TEXT NOT NULL,
    assigned_at     TIMESTAMPTZ DEFAULT NOW(),
    released_at     TIMESTAMPTZ,
    UNIQUE(session_id)
);
```

> **Critical redesign note:** The existing 
elay_instances table must be redesigned or retired for shared relays. Currently, ActivateProvisionedSession() always inserts a brand-new 
elay_instances row with a unique ws_instance_id and unique session_id � this is fundamentally incompatible with multiple sessions sharing one relay host. Similarly, RecordRelayHealth() validates heartbeats by joining sessions to 
elay_instances on 
i.aws_instance_id, which breaks when many sessions map to one server. The 
elay_pool + 
elay_assignments tables should become the sole source of truth, and 
elay_instances should be phased out for the shared-pool path. Keeping both models will create contradictory sources of truth.

The 
elay_pool table is manually populated — Hetzner servers are provisioned once (manually or via a setup script), then registered in the pool. This is NOT dynamic auto-scaling; it's a fixed pool that grows by manually adding servers.

### 3.2 HetznerProvisioner

**Estimated effort:** 1–1.5 weeks (revised upward: the hard part is replacing the session/relay_instances model, not the provisioner itself)

```go
type HetznerProvisioner struct {
    store     *store.Store
    sls       *SLSClient  // HTTP client for SLS management API
}

func (p *HetznerProvisioner) Provision(ctx context.Context, req ProvisionRequest) (ProvisionResult, error) {
    // 1. Find an available relay server in the pool
    server, err := p.store.AssignRelay(req.UserID, req.SessionID, req.Region, req.StreamToken)
    // AssignRelay picks: region-affinity first, then least-loaded, then any available
    
    // 2. Create user's stream IDs on the shared relay via SLS API
    err = p.sls.CreateStreamIDs(server.Host, req.StreamToken)
    // POST http://{server.host}:8090/api/stream-ids
    // Creates live_{stream_token} and play_{stream_token}
    
    // 3. Return the shared server's connection details
    return ProvisionResult{
        InstanceID:   server.ServerID,
        PublicIP:     server.IP,
        SRTPort:      5000,
    }, nil
}

func (p *HetznerProvisioner) Deprovision(ctx context.Context, req DeprovisionRequest) error {
    // 1. Remove user's stream IDs from the relay
    err := p.sls.DeleteStreamIDs(server.Host, streamToken)
    
    // 2. Release the assignment (decrement current_sessions)
    err = p.store.ReleaseRelay(req.SessionID)
    
    // 3. Do NOT destroy the server — it stays in the pool
    return nil
}
```

**Key difference from AWS flow:**
- No instance launch/terminate. The server is already running.
- No EIP allocation. The shared server has a fixed IP.
- No user-data script. The Docker stack is pre-deployed.
- Provision is fast (HTTP call to SLS API, <1 second) vs AWS (60–90 seconds for EC2 boot).
- The provision step tracking (`launching_instance` → `waiting_for_instance` → ...) can be simplified or skipped.

### 3.4 Pool Health Monitor

**Estimated effort:** 1 day

Background goroutine that runs on a timer (every 60 seconds):
- Iterates over all `active` servers in `relay_pool`
- Calls `SLSClient.HealthCheck()` for each
- Updates `last_health_check` and `health_status`
- If a server is `unhealthy` for >3 consecutive checks, marks it `draining` (no new assignments)
- Alerts (log + optional webhook) if a server goes offline

### 3.5 Assignment Logic

**Estimated effort:** 0.5 days

`store.AssignRelay(userID, sessionID, region, streamToken)`:

1. Query `relay_pool` where `status = 'active'` and `current_sessions < max_sessions` and `health_status = 'healthy'`
2. Filter by region affinity: prefer same region as user's request
3. Sort by `current_sessions ASC` (least-loaded first)
4. If no servers available in preferred region, fall back to any region
5. If no servers available at all, return `ErrNoRelayCapacity`
6. Atomically: increment `current_sessions`, insert `relay_assignments` row
7. Return the selected server

### 3.6 Provision Pipeline Adaptation

**Estimated effort:** 0.5 days

The existing `runProvisionPipeline()` in `handlers.go` runs 6 steps with 3-second dwells. For Hetzner shared relays, most of those steps are instant:

- `launching_instance` → skip (server already running)
- `waiting_for_instance` → skip
- `starting_docker` → skip (containers already running)
- `starting_containers` → skip
- `creating_stream` → this is the only real step (SLS API call)
- `ready` → done

Options:
- **Option A:** Fast-path the pipeline: if provider is Hetzner, jump straight from `creating_stream` to `ready` with no artificial dwells.
- **Option B:** Keep the step progression but with 0-second dwells, so the dock UI flashes through quickly. This gives a consistent UX.

**Recommendation:** Option A. Users will appreciate the instant start.

### 3.7 No Per-User EIP

Shared Hetzner relays have a fixed IP. No per-user EIP needed. The `StaticIPManager` interface from Phase 1 simply isn't implemented by `HetznerProvisioner` — the pipeline skips it automatically.

DNS records (`{slug}.relay.telemyapp.com`) point to the shared relay's IP. Multiple users can share the same relay IP because stream isolation is handled by unique stream tokens, not by separate IPs.

### 3.8 Noisy Neighbor Mitigation

**Estimated effort:** 1 day (alongside pool health monitor work)

Shared relays introduce contention risk. One user's high-bitrate or malformed stream can degrade quality for other users on the same node.

**Mitigations (implement at launch):**
- **Strict max_sessions limits per node:** Start conservatively at 10 sessions per CX22 (not the 20 in the schema default). Increase only after load testing validates headroom.
- **Per-container CPU/network monitoring:** Track CPU and network utilization per srtla_rec and SLS process. Alert (log + webhook) if a single user's stream exceeds expected resource consumption.
- **Tier-based density:** Pro tier gets lower contention (e.g., 5 users/node) vs Starter (10 users/node). Implement via separate max_sessions values or dedicated pool segments per tier.
- **Kill switch:** Ability to instantly revoke a specific user's stream token on a shared relay if they are causing degradation (via SLS API DELETE /api/stream-ids/live_{token}).

### 3.9 Admin Dashboard for Relay Pool Management

**Estimated effort:** 1�2 days

A simple admin tool (web UI or CLI) for managing the shared relay pool:
- **Pool overview:** List all servers with status, current sessions, health, region
- **Drain/reboot:** Mark a server as draining (no new assignments), wait for sessions to end, then reboot
- **Server status:** View per-server metrics (CPU, network, active streams)
- **Kill switch:** Terminate a specific user's stream on a specific server
- **Add/remove servers:** Register new Hetzner servers into the pool or decommission old ones

This can start as a CLI tool (egis-admin pool list, egis-admin pool drain <server_id>) and evolve into a web UI later.

### 3.10 Phase 3 Deliverable

At the end of Phase 3, a paid user can:
1. Click "Start Relay" in the dock
2. Get assigned to a shared Hetzner relay in <2 seconds (vs 60–90s on AWS)
3. Stream via `srtla://{relay_ip}:5000` with `streamid=live_{stream_token}`
4. See full stats in the dock (same as v0.0.4)
5. Click "Stop Relay" — stream IDs are cleaned up, server stays running

## Phase 4: Control Plane Migration (Week 3–4)

**Goal:** Move the aegis-api Go binary from AWS EC2 (`<redacted-ec2-ip>`) to Hetzner Cloud. Reduce baseline hosting cost and consolidate on one provider.

### 4.1 Hetzner Server Setup

**Estimated effort:** 0.5 days

- Provision a Hetzner CX22 (2 vCPU, 4GB RAM, 40GB disk) — ~€4.35/mo (~$5/mo)
- Or co-locate on a relay pool server if capacity allows (saves ~$5/mo but introduces coupling)
- **Recommendation:** Separate box. Control plane availability shouldn't depend on relay health.
- Install: Go runtime (or deploy static binary), PostgreSQL (or connect to managed Postgres), systemd service, certbot for TLS

### 4.2 Database Migration

**Estimated effort:** 1 day

> **Note:** The control plane uses PostgreSQL via `pgx`, NOT SQLite. All existing migrations use Postgres types (`inet`, `timestamptz`, partial indexes, `uuid`). The migration strategy must account for this.

Migration plan:
1. Provision PostgreSQL on the Hetzner server (or use a managed Postgres service)
2. Stop the control plane on EC2
3. Export the database: `pg_dump -Fc -h <EC2_DB_HOST> -U <DB_USER> telemy > telemy_backup.dump`
4. Restore to the new Postgres instance: `pg_restore -h <HETZNER_DB_HOST> -U <DB_USER> -d telemy telemy_backup.dump`
5. Update the connection string in the Hetzner deployment config (`DATABASE_URL` or equivalent `pgx` config)
6. Verify all migrations applied: check `schema_migrations` table
7. Verify row counts match between old and new databases
8. Start the control plane on Hetzner
9. Verify all auth endpoints, entitlements, relay lifecycle, session state

**Rollback:** Keep the EC2 Postgres instance running for 48 hours. If issues arise, revert DNS and connection strings to the original database.

### 4.3 DNS Cutover

**Estimated effort:** 1 hour (but with monitoring)

- Update Cloudflare A record: `api.telemyapp.com` → new Hetzner IP
- Set low TTL (60s) a day before the migration, then cut over
- Monitor for errors on both old and new servers during propagation
- Keep the EC2 instance running for 48 hours as a fallback (reverse DNS if needed)
- After 48 hours with no issues, terminate the EC2 instance

### 4.4 Config Cleanup

- Remove `AEGIS_AWS_*` env vars from the Hetzner deployment (not needed if default provider is `hetzner`)
- Keep AWS credentials in a config file (not env vars) for backward compat — the `AWSProvisioner` code stays, it just isn't the active provider
- Update systemd service file with new environment

## Phase 6: Video Stabilization (v0.0.6 � After Phases 1-5a are shipped)

> **Explicitly scoped to v0.0.6.** Video stabilization is a client-side R&D project in a completely different domain (Rust/wgpu/compute shaders) from the infrastructure/business model work in v0.0.5. Build the sustainable business model first, then build the killer feature to drive upgrades.

Real-time video stabilization as an optional OBS filter for IRL streamers. This is a differentiator � no competitor offers this.

Full design doc: `docs/future/video-stabilization.md`

### Summary

Software-based stabilization running on the receiving PC's GPU (compute shaders via Rust + wgpu), processing the incoming SRT/SRTLA feed after arrival and before OBS encode. Local-only � cloud-hosted stabilization was evaluated and rejected (latency, bandwidth, cost).

### Key Design Decisions

- **Build fresh in Rust with wgpu** � do NOT fork LiveVisionKit (OpenCV/OpenCL driver issues, different codebase conventions). Use LVK as algorithm/UX reference only.
- **Completely decoupled** from core relay/failover logic � independent optional filter
- **Hardware gated** � minimum RTX 2060 / RX 5000, runtime VRAM and compute utilization checks, auto-disable above ~75% GPU compute usage
- **Configurable crop factor** � slider 1-15%, sensible defaults (3-5% handheld, 8-10% walking)
- **Audio sync** � Rust core handles automatic delay compensation for frame buffer latency
- **Suppression mode** � auto-disables during intentional rapid panning

### Processing Pipeline

```
Incoming frame (SRT/SRTLA source)
  ? Motion Estimation (compute shader)
  ? Smoothing Buffer (N frames)
  ? Corrective Transform (compute shader)
  ? Crop & Scale (+ optional FSR sharpening)
  ? Output ? OBS encode
```

### Resource Impact

- Uses GPU compute cores (NOT the hardware encoder � NVENC/AMF runs in parallel)
- ~200-400MB VRAM at 1080p
- Excellent on dedicated encoding PCs (RTX 2060+), good with dual H/V output
- Not recommended for GTX 1060 or below

### Implementation Effort

- Estimated 4-6 weeks for MVP (motion estimation + basic smoothing + crop)
- Prototype first: minimal wgpu compute shader for frame-to-frame motion estimation
- Benchmark across GPU generations before committing to approach

### Open Questions (Resolve at Implementation Time)

1. Motion estimation algorithm: optical flow vs phase correlation vs block matching
2. OBS integration point: register as OBS filter or process within aegis-core
3. Gyroscope metadata from phones for hybrid stabilization
4. Interaction with OBS scene transforms and crop filters

### Why This Matters for Telemy

IRL streamers are the core audience. Shaky footage is their #1 quality complaint. Current solutions (gimbals, in-camera EIS) all have gaps. A bundled stabilizer in the Telemy plugin � alongside failover, relay, telemetry � makes Telemy the complete IRL toolkit, not just a relay manager.

## Open Questions

### 1. Should BYOR users need an account at all?

**Arguments for requiring an account:**
- Telemetry sync across sessions (useful for the user)
- Usage analytics for Telemy (useful for us)
- Upsell path from free to paid
- Scene preferences need server-side storage if syncing across machines

**Arguments against:**
- Lower friction for adoption
- Some users just want the OBS dock widget, no cloud dependency
- Reduces control plane load

**Proposed answer:** Account optional. Plugin works fully offline in BYOR mode (local config, local scene prefs). If the user creates an account, their BYOR config syncs to the server and they get cross-machine persistence. This is the best of both worlds.

### 2. How to handle BYOR users' relay stats if their relay doesn't run SLS?

**Proposed answer:** Graceful degradation. The plugin attempts to poll `:8090/stats` and `:5080/stats`. If either returns a connection error or non-200 status for 3 consecutive polls, that section is hidden in the dock with a subtle "Stats unavailable for custom relays" message. The rest of the dock (OBS telemetry, system stats, scene management) works regardless.

For users running Belabox: Belabox uses SLS internally, so stats *might* work if the stats port is exposed. Worth testing.

### 3. Minimum viable Hetzner regions to launch with?

**Proposed answer:** Start with two:
- **US East (Ashburn)** — Hetzner's `ash` datacenter, covers East Coast and Central US
- **US West (Hillsboro)** — Hetzner's `hil` datacenter, covers West Coast

Most Telemy users are US-based IRL streamers. EU can be added later with Hetzner's `fsn1` (Falkenstein) or `nbg1` (Nuremberg).

Assignment logic defaults to `us-east` if no region preference, or picks based on user's last known location.

### 4. Whether to publish the srtla-receiver Docker stack as a public self-host guide

**Proposed answer:** Yes, but after v0.0.5 launches and BYOR is validated. The Docker image is already public (`ghcr.io/michaelpentz/srtla-receiver:latest`). A simple README with:
- `docker-compose.yml` example
- Firewall rules (UDP 5000, TCP 4000, TCP 8090, TCP 5080)
- Stream ID creation via SLS API
- Telemy plugin BYOR configuration

This increases adoption (more BYOR users → more potential conversions to paid) and positions Telemy as the ecosystem tool for IRL streaming, not just a relay provider.
